# RipCurl 3.0.0: Collapse handler SPI + customizer-only extension

**Goal:** Remove the `JsonRpcMethod` / `JsonRpcMethodProvider` / `AnnotationJsonRpcMethod` indirection and the `@JsonRpcService` class-level marker. Handlers are built directly from `(bean, @JsonRpcMethod method)` pairs via a pure-Java helper. All per-handler extension — resolvers and interceptors — flows through a single `JsonRpcMethodHandlerCustomizer` SPI. The bean-level autowiring paths `List<ParameterResolver<? super JsonNode>>` and `List<MethodInterceptor<? super JsonNode>>` (both present in 2.7.0) are removed. There is exactly one path for contributing either a resolver or an interceptor to a handler, and it is the customizer.

**Why:**

1. **Remove speculative surface area.** The `JsonRpcMethod` / `JsonRpcMethodProvider` SPI was added to support programmatic non-annotated handler registration. No one does this. Collapsing it to one concrete `JsonRpcMethodHandler` record and one builder eliminates six types.

2. **Eliminate the "every matching bean gets wired in" footgun.** 2.7.0's bean-list paths for both resolvers and interceptors were structural coincidences — any `ParameterResolver<? super JsonNode>` or `MethodInterceptor<Object>` bean from any source (another starter, a test, a transitive dep) joined every RipCurl handler's pipeline. The customizer SPI makes contributions intentional: you implement `JsonRpcMethodHandlerCustomizer` and register it as a bean.

3. **Per-handler context at attachment time.** Customizers receive `(name, method, bean)` when they decide what to attach. Interceptors can bake per-handler state (method-name tag, annotation metadata, auth predicate) into their closure at construction instead of re-introspecting on every invocation.

4. **Mirror the Mocapi cleanup** (specs 172, 177, 180). Same shape, same SPI ergonomics. Consistent model across the Callibrity JSON-RPC / MCP stack.

**Breaking changes:**

- `@JsonRpcService` class-level annotation removed. Users annotate methods with `@JsonRpcMethod`; the bean itself becomes a Spring bean via `@Component` (or however the app normally registers it).
- `JsonRpcMethod` SPI interface removed.
- `JsonRpcMethodProvider` SPI interface removed.
- `AnnotationJsonRpcMethod`, `AnnotationJsonRpcMethodProviderFactory`, `DefaultAnnotationJsonRpcMethodProviderFactory` removed.
- `JsonRpcServiceMethodProvider` removed.
- `DefaultJsonRpcDispatcher` constructor now takes `List<JsonRpcMethodHandler>` instead of `List<JsonRpcMethodProvider>`.
- `RipCurlResolversAutoConfiguration` deleted; `JsonRpcParamsResolver` stops being a registered Spring bean. It and `Jackson3ParameterResolver` are constructed inline inside `JsonRpcMethodHandlers.build(...)` as the baked-in head and tail of every handler's resolver chain.
- Bean-level autowiring of `List<ParameterResolver<? super JsonNode>>` and `List<MethodInterceptor<? super JsonNode>>` in `RipCurlAutoConfiguration` is gone. Both resolver and interceptor contributions are customizer-only.

### Add

**ripcurl-core — `core/annotation/JsonRpcMethodHandler.java`:**

```java
public final class JsonRpcMethodHandler {

  private final String name;
  private final Method method;
  private final Object bean;
  private final MethodInvoker<JsonNode> invoker;
  private final ObjectMapper mapper;

  JsonRpcMethodHandler(
      String name, Method method, Object bean,
      MethodInvoker<JsonNode> invoker, ObjectMapper mapper) {
    this.name = name;
    this.method = method;
    this.bean = bean;
    this.invoker = invoker;
    this.mapper = mapper;
  }

  public String name() { return name; }
  public Method method() { return method; }
  public Object bean() { return bean; }

  public JsonRpcResult call(JsonRpcRequest request) {
    var result = invoker.invoke(request.params());
    if (result instanceof JsonRpcResult jsonRpcResult) {
      return jsonRpcResult;
    }
    JsonNode id = request instanceof JsonRpcCall call ? call.id() : null;
    JsonNode jsonResult = result == null ? NullNode.getInstance() : mapper.valueToTree(result);
    return new JsonRpcResult(jsonResult, id);
  }
}
```

Concrete, no interface, no SPI. `call(...)` body is moved verbatim from the old `AnnotationJsonRpcMethod`.

**ripcurl-core — `core/annotation/JsonRpcMethodHandlers.java`:**

Pure-Java builder. Public static `build(bean, method, ...)`, no Spring imports.

```java
public final class JsonRpcMethodHandlers {

  private JsonRpcMethodHandlers() {}

  public static JsonRpcMethodHandler build(
      Object bean,
      Method method,
      ObjectMapper mapper,
      MethodInvokerFactory invokerFactory,
      List<JsonRpcMethodHandlerCustomizer> customizers) {
    var annotation = method.getAnnotation(JsonRpcMethod.class);
    String className = ClassUtils.getSimpleName(bean);
    String methodName = method.getName();
    String name = ofNullable(StringUtils.trimToNull(annotation.value()))
        .orElseGet(() -> String.format("%s.%s", className, methodName));

    var config = new MutableConfig(name, method, bean);
    customizers.forEach(c -> c.customize(config));

    var paramsResolver = new JsonRpcParamsResolver(mapper);
    var jacksonResolver = new Jackson3ParameterResolver(mapper);
    var userResolvers = config.freezeResolvers();
    var interceptors = config.freezeInterceptors();

    var invoker = invokerFactory.create(
        method, bean, JsonNode.class,
        cfg -> {
          cfg.resolver(paramsResolver);        // built-in head
          userResolvers.forEach(cfg::resolver); // customizer-added, in order
          cfg.resolver(jacksonResolver);        // built-in tail
          interceptors.forEach(cfg::interceptor);
        });

    return new JsonRpcMethodHandler(name, method, bean, invoker, mapper);
  }
  // MutableConfig tracks resolvers and interceptors separately; both frozen at build time.
}
```

Both built-in resolvers are constructed inline every call — they're cheap stateless wrappers around the `ObjectMapper`. The builder enforces chain ordering: nothing can insert before `JsonRpcParamsResolver` or after `Jackson3ParameterResolver`. Methodical's `@Argument` tail still runs after the Jackson catch-all.

### Modify

**ripcurl-autoconfigure — `RipCurlAutoConfiguration`:**

Replace `JsonRpcServiceMethodProvider` + `AnnotationJsonRpcMethodProviderFactory` + the interceptor-collecting bean. Handler discovery becomes an inline one-pass scan over the bean factory — no separate cache type is needed since RipCurl only has one handler annotation to group by (unlike Mocapi, which scans for four and caches to avoid repeating the walk).

The scan uses `ConfigurableListableBeanFactory.getBeanNamesForType(Object.class, false, false)` + `getType(name, false)` — the `allowEagerInit=false` forms — so lazy / prototype / `FactoryBean` beans aren't instantiated just to find out they don't host `@JsonRpcMethod`. Two unwraps: `ClassUtils.getUserClass(declaredType)` for the pre-check against the declared type, then `AopUtils.getTargetClass(bean)` once we know we need the bean. Annotations on proxied handler classes (e.g., `@Transactional` on a handler) stay discoverable on the unwrapped target class; invocation still goes through the proxy so advice runs.

```java
@Bean
List<JsonRpcMethodHandler> jsonRpcMethodHandlers(
    ConfigurableListableBeanFactory beanFactory,
    ObjectMapper mapper,
    MethodInvokerFactory invokerFactory,
    List<JsonRpcMethodHandlerCustomizer> customizers) {
  var handlers = new ArrayList<JsonRpcMethodHandler>();
  for (String name : beanFactory.getBeanNamesForType(Object.class, false, false)) {
    var declaredType = beanFactory.getType(name, false);
    if (declaredType == null) {
      continue;
    }
    var userType = ClassUtils.getUserClass(declaredType);
    if (MethodUtils.getMethodsListWithAnnotation(userType, JsonRpcMethod.class).isEmpty()) {
      continue;
    }
    var bean = beanFactory.getBean(name);
    var targetClass = AopUtils.getTargetClass(bean);
    for (var method : MethodUtils.getMethodsListWithAnnotation(targetClass, JsonRpcMethod.class)) {
      handlers.add(JsonRpcMethodHandlers.build(
          bean, method, mapper, invokerFactory, resolvers, customizers));
    }
  }
  return List.copyOf(handlers);
}

@Bean
public JsonRpcDispatcher jsonRpcDispatcher(
    List<JsonRpcMethodHandler> handlers,
    JsonRpcExceptionTranslatorRegistry translators) {
  return new DefaultJsonRpcDispatcher(handlers, translators);
}
```

`List<MethodInterceptor<? super JsonNode>>` is no longer a bean dependency anywhere.

**ripcurl-core — `DefaultJsonRpcDispatcher`:**

Constructor takes `List<JsonRpcMethodHandler>` instead of `List<JsonRpcMethodProvider>`. Build a `Map<String, JsonRpcMethodHandler>` once in the constructor (eager, not lazy — the handler list is fully built before the dispatcher bean is constructed). Drop the `LazyInitializer` indirection entirely; dispatch does a direct `methods.get(name)` lookup.

**ripcurl-jakarta-validation — new `JakartaValidationCustomizer`:**

```java
public final class JakartaValidationCustomizer implements JsonRpcMethodHandlerCustomizer {

  private final JakartaValidationInterceptor interceptor;

  public JakartaValidationCustomizer(JakartaValidationInterceptor interceptor) {
    this.interceptor = interceptor;
  }

  @Override
  public void customize(JsonRpcMethodHandlerConfig config) {
    config.interceptor(interceptor);
  }
}
```

Unconditional apply-to-all. A per-method "has any `jakarta.validation.*` annotation" filter is a premature optimization; validation's cost on an unannotated method is already trivial.

Registered as a bean by `RipCurlJakartaValidationAutoConfiguration`, conditional on the existing `@ConditionalOnClass({ConstraintViolationException.class, ConstraintViolationExceptionTranslator.class})` plus a new `@ConditionalOnBean(JakartaValidationInterceptor.class)` so it only fires when Methodical's jakarta autoconfig is also active.

### Tests

Expected changes:

- **Delete** any test that exercised `AnnotationJsonRpcMethod` / `JsonRpcMethodProvider` directly — redundant coverage of removed types.
- **Migrate** all `@JsonRpcService`-annotated test fixtures to `@Component` (or explicit `@Bean` where the test fixture isn't component-scanned).
- **Unit tests for `JsonRpcMethodHandlers.build`** — pure-Java construction, no Spring. Assert handler name derivation (annotation value vs. default), resolver ordering in the built invoker, customizer invocation order, and interceptor chain ordering.
- **Unit tests for `JsonRpcMethodHandlerCustomizer`** — register a stub customizer, assert it received a config with the right `name`/`method`/`bean`, and assert the interceptor it attached ran during a simulated invocation.
- **Integration test for `JakartaValidationCustomizer`** — existing validation → `-32602` end-to-end test should pass unchanged if the customizer is correctly wired.
- **Tests that passed raw `MethodInterceptor` beans** to assert chain participation convert to customizers. Semantically equivalent, new SPI shape.

## Migration guide (for CHANGELOG and release notes)

### Handler classes

```diff
-@JsonRpcService
+@Component
 public class Calculator {
     @JsonRpcMethod("subtract")
     public int subtract(int a, int b) { return a - b; }
 }
```

Classes registered via `@Bean` rather than `@Component` scanning need no change.

### Custom interceptors

```diff
-@Bean
-MethodInterceptor<JsonNode> timingInterceptor() {
-  return invocation -> { /* ... */ };
-}
+@Bean
+JsonRpcMethodHandlerCustomizer timingCustomizer() {
+  return config -> config.interceptor(invocation -> { /* ... */ });
+}
```

A customizer can read `config.name()` / `config.method()` / `config.bean()` to decide whether to attach the interceptor at all — behavior that was impossible via the bean-list path.

### Custom resolvers

No change. Resolver contributions still flow via `ParameterResolver<? super JsonNode>` beans — resolvers are per-invoker infrastructure, not per-handler extension.

### Programmatic handler registration

If anyone was implementing `JsonRpcMethodProvider` or `JsonRpcMethod` directly, that path is gone. Drop down to the `JsonRpcMethodHandlers.build(...)` helper directly or produce a `List<JsonRpcMethodHandler>` bean via your own factory. The `JsonRpcMethodHandler` constructor is package-private; builders go through the helper.

## AOT / native-image impact

The `JsonRpcServiceBeanAotProcessor` is keyed on `@JsonRpcService`. After this spec, it keys on "any bean with at least one `@JsonRpcMethod` method" and must use the same `AopUtils.getTargetClass(bean)` + `MethodUtils.getMethodsListWithAnnotation(..., JsonRpcMethod.class)` walk as the runtime scan. The reflection hints it emits are per-method, so the switch is mechanical.

`RipCurlRuntimeHints` loses its `@JsonRpcService` reference. No change to the message-type hints.

The jakarta-validation module's runtime hints don't change shape — the interceptor bean is still there, it's just contributed through a customizer now.

## Non-goals

- **Reworking the exception translator SPI.** It's not touched. `JsonRpcExceptionTranslator` / `JsonRpcExceptionTranslatorRegistry` are already the right shape.
- **Changing the parameter resolver story.** Resolvers remain bean-collected; they're per-invoker infrastructure, not per-handler extension.
- **Hot-reload / dynamic handler registration.** Discovery is one-pass at startup, same as today.
- **A customizer ordering API.** Spring's bean-ordering (`@Order`, `Ordered`) applies to the autowired `List<JsonRpcMethodHandlerCustomizer>`. No new ordering surface.
- **Per-method customizer filtering via annotations** (e.g., "only apply this customizer when the method has `@SomeAnnotation`"). Customizers can filter themselves by reading `config.method()`.
